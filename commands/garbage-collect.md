# Garbage Collect

Background sweep that scans for drift against `docs/spec/GOLDEN_PRINCIPLES.md` and proposes targeted cleanup. Modeled on the harness-engineering pattern: small, continuous refactor pressure beats periodic debt sprints.

## Usage

```
/garbage-collect                       # Dry-run scan, print a grouped report
/garbage-collect --report              # Write the report to docs/plan/GC_REPORT_<date>.md
/garbage-collect --principle GP-002    # Scope to a single rule
/garbage-collect --since HEAD~50       # Only scan files changed in the last 50 commits
/garbage-collect --fix                 # Apply auto-fix hints (creates a branch per principle)
/garbage-collect --open-prs            # Imply --fix and push branches + open PRs
```

Default is dry-run + print. `--fix` and `--open-prs` require explicit opt-in — they create branches and (with `--open-prs`) push to remote.

## Process

### 1. Load the rule set

Read `docs/spec/GOLDEN_PRINCIPLES.md`. Parse each `## GP-NNN` section into:
- ID
- One-line rule statement
- Auto-fix hint

Skip any rule marked `RETIRED`. If `--principle` is set, filter to that ID.

### 2. Determine the scan set

Default: all tracked source files (exclude generated paths, lockfiles, vendored directories — see `.gitignore` and check `docs/spec/LAYERS.md` mappings for generated patterns).

If `--since <ref>` is set: only files changed in `git diff --name-only <ref>...HEAD`.

### 3. Dispatch a scan agent per principle

Dispatch one subagent per GP rule (use the `Agent` tool, `Explore` subagent type, run in parallel — cap at 3 concurrent). Each agent gets:

| Field | Value |
|-------|-------|
| **Files to read** | The scan set (read-only) + `docs/spec/GOLDEN_PRINCIPLES.md` (for the rule context) + relevant spec files |
| **Files to modify** | None (scan-only in dry-run) |
| **Work** | Find concrete violations of GP-NNN in the scan set. For each, capture: file:line, a one-line description, severity (critical / high / medium / low) |
| **Completion criteria** | Returns a structured list of violations (or "no violations found") |
| **Scope guard** | Do not propose fixes in this phase — only identify |

**Scan heuristics by rule** (the coordinator should hand these to each agent):

- **GP-001** (centralize invariants): grep for near-duplicate helper functions — same name across modules, same signature with renamed params, repeated regex/format strings.
- **GP-002** (validate at boundaries): grep for `.get(...).get(` chains, repeated `isinstance` ladders, request/response handlers that don't reference a schema class.
- **GP-003** (one source of truth): grep for repeated literal constants, parallel enum definitions, hard-coded route paths that exist in routing config.
- **GP-004** (actionable errors): grep for `raise Exception(`, `throw new Error(`, error messages without context strings.
- **GP-005** (deterministic tests): grep in test directories for `datetime.now()`, `Date.now()`, `time.time()`, unmocked `fetch`/`requests`/`http.get`, `time.sleep` calls, tests that depend on file order.
- **GP-006** (one-way deps): defer to `/check-layers` — note this and skip.
- **GP-007** (typed public APIs): grep for `def f(` without type hints on parameters (Python), unexported functions without TypeScript annotations, `Any`/`unknown` in public signatures.
- **GP-008** (no dead code): grep for `//\s*(TODO|FIXME|removed|deprecated|unused)`, large commented-out blocks, exports that have no importers.
- **GP-009** (signal-routed side effects): grep for cross-module imports where module A's call into module B has no signal/event layer in between (heuristic: A importing B in patterns that match circular-import risk).
- **GP-010** (composition over inheritance): grep for inheritance chains deeper than 2 levels, classes with only one subclass.
- **GP-011** (typed config): grep for `os.environ`, `os.getenv`, `process.env.` outside the config module.
- **GP-012** (regenerate generated code): check for hand-edits in files marked generated (look for "AUTOGENERATED" / "DO NOT EDIT" banners with recent non-generator commits touching them).

### 4. Aggregate and rank

Coordinator merges the per-agent reports into a single grouped report:

```markdown
# Garbage Collect Report — <date>

## Summary
- GP-001: 7 violations (1 critical, 3 high, 3 medium)
- GP-002: 12 violations (4 high, 8 medium)
- GP-003: 2 violations (medium)
- GP-008: 23 violations (low)
- ...
- Total: <N> violations across <M> rules

## GP-001 — Centralize invariants in shared utilities
### Critical
- backend/users/helpers.py:42 — `format_user_name()` duplicates `backend/utils/formatting.py:18` (3 lines drift)
### High
- ...

## GP-002 — Validate at boundaries
...
```

### 5. Report or fix

**Dry-run (default):** Print the report. If `--report` is set, also write to `docs/plan/GC_REPORT_<YYYY-MM-DD>.md`.

**`--fix`:** Before applying any fixes, enforce the coverage guard:

1. **Coverage gate** (mechanical — calls `lib/validate.py:check_coverage_gate()`):
   - Read `vibe.config.json` for `garbage_collect.min_coverage_for_fix` (default: 70) and `garbage_collect.coverage_command` (inferred from stack if absent).
   - Run the coverage command to get the current coverage percentage.
   - Call `check_coverage_gate(min_coverage, actual_coverage)`.
   - If the gate returns `passed=False`: print the reason string, write the scan report to `docs/plan/GC_REPORT_<date>.md`, and **stop**. Do not apply fixes. The reason string includes instructions for the `--force` override.
   - If `--force` is also passed: log `force-used: true` in `GC_LOG.md`, print a warning, and proceed despite low coverage.

2. For each principle with violations:
   a. Create a branch: `git checkout -b gc/<date>/GP-NNN`.
   b. Dispatch the `garbage-collector` subagent with:
      - **Files to read:** violations list, GP rule + auto-fix hint, `docs/spec/CONVENTIONS.md`
      - **Files to modify:** only files listed in the violations for this GP rule
      - **Work:** apply the auto-fix hint to each violation
      - **Completion criteria:** all violations addressed, test suite passes
      - **Scope guard:** only modify files in the violations list for this GP-NNN
   c. If tests pass: commit with `refactor(GP-NNN): garbage-collect sweep — <count> violations`.
   d. If tests fail: leave the branch with the partial work and a `GC_FAILED_GP-NNN.md` note in `docs/plan/`. Skip to next principle.

**`--open-prs`:** Imply `--fix`. After each successful branch, push it and open a PR with:
- Title: `refactor(GP-NNN): garbage-collect sweep — <count> violations`
- Body: the violations list + auto-fix hint + test results

### 6. Update the active sweep log

Append a line to `docs/plan/GC_LOG.md` (create if missing):

```markdown
| Date | Mode | Violations found | Branches opened | Notes |
|------|------|------------------|-----------------|-------|
| 2026-05-12 | dry-run | 47 across 8 GPs | 0 | |
| 2026-05-13 | --fix   | 12 across 3 GPs | 3 | GP-011 failed: env access in test fixtures |
```

This log gives the team a longitudinal view of where drift accumulates.

## Running on a schedule

The harness-engineering article emphasizes that garbage collection is *continuous*, not periodic. Two patterns:

**Manual cadence:** Run `/garbage-collect --report` weekly to track drift. Run `/garbage-collect --fix --principle GP-NNN` when one rule accumulates enough violations to warrant a sweep.

**`/loop` integration:** Tools like `/loop` can run this on an interval (e.g., `/loop 1d /garbage-collect --report`). Background sweeps mean drift is caught while it's small.

## Configuration (`vibe.config.json`)

The coverage guard reads from the `garbage_collect` section of `vibe.config.json`:

```json
{
  "garbage_collect": {
    "min_coverage_for_fix": 70,
    "coverage_command": "python -m pytest --cov --cov-report=term-missing"
  }
}
```

| Key | Default | Description |
|-----|---------|-------------|
| `min_coverage_for_fix` | `70` | Minimum test coverage percentage required before `--fix` applies changes. Below this, only a report is generated. Override with `--force`. |
| `coverage_command` | *(inferred from stack)* | Shell command that produces coverage output. If absent, inferred from `vibe.config.json` backend/frontend test commands. |

## Anti-patterns to avoid

- **Mass `--fix` without test coverage.** If the test suite is weak, `--fix` will silently degrade behavior. Run `--report` first; expand tests for the affected modules; then run `--fix`.
- **Fixing across principles in one branch.** Each branch should target exactly one GP-NNN so reviews stay focused and reversible.
- **Treating low-severity violations as blockers.** Low items are tracked for trend analysis, not gated on. Critical and high items get sweeps; medium and low accumulate until a rule's count justifies a pass.
- **Hand-editing generated files in the sweep.** GP-012 violations are fixed by re-running the generator, not by patching the output.

## Output contract

`/garbage-collect` MUST:
1. Print a one-line summary at the top: `Found <N> violations across <M> GP rules. <K> critical.`
2. Exit non-zero if any critical violations were found in `--report` mode (so CI can gate on it).
3. Never modify a file outside an explicit `--fix` or `--open-prs` invocation.
